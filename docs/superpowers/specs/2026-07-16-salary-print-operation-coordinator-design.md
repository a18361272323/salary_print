# 工资表打印操作协调与列配置批量保存设计

## 目标

把“保存列配置”从浏览器按列循环调用模型方法，改为一次 API 编排调用。所有需要等待接口的操作共享一个可见、互斥的操作状态，防止用户在请求期间改变当前薪资组、所属期或列配置快照。

本次仅改造列配置写入；版式配置仍使用现有模型方法，但同样接入操作协调器。

## 范围

包含：初始化薪资组、工资表查询、保存列配置、保存版式、打印准备的加载状态与交互锁。

不包含：改变列配置模型字段、改变版式配置模型字段、改变工资数据来源、在浏览器中直接批量写模型。

## 操作协调器

前端维护唯一活动操作：`idle`、`initializing`、`querying`、`savingColumns`、`savingLayout`、`preparingPrint`。

开始操作时，协调器记录不可变快照（薪资组、所属期、查询令牌）、展示全局状态，并锁定会改变该快照或配置草稿的控件：查询、薪资组/所属期切换、纸张切换、列勾选、拖拽、重置、保存列配置、编辑版式、保存版式和打印。页面阅读与滚动不锁定。

结束时，无论成功或失败，协调器都恢复控件。查询沿用已加载条数进度；保存类操作显示不定进度和保存数量。保存进行中触发页面离开时显示离开提示，不试图在浏览器关闭期间补发写请求。

## 保存列配置 API 编排

新增一个仅用于当前打印列配置的 API 编排。浏览器一次提交完整列配置快照：薪资组、所属期、列数组和调用版本标识。调用方不传可信用户身份；编排从当前登录态解析配置人。

编排在同一事务内完成：

1. 校验薪资组、所属期、列键非空且不重复，显示顺序合法，布尔字段和分组字段符合当前列配置契约。
2. 查询当前配置人的同薪资组、所属期、`profile_type=column` 的已有记录。
3. 按 `column_key` 新增或更新本次完整快照中的记录。
4. 返回最终记录、保存数量和服务端版本；任一写入失败则整个事务失败，不返回部分成功。

API 输入、输出、模型方法标识、事务节点可用性和当前登录态变量形态均需在目标 API 编辑器回读确认后生成。浏览器调用遵循现有同源 API 调用方式，响应以 `{ returnCode, errorMsg, body }` 校验。

前端成功后直接以 API 返回记录更新 `preferenceRecords`，不再立即追加一次列配置列表回读；后续查询仍从模型读取最新记录。

## 错误与并发

活动操作期间的新操作被拒绝并保留当前状态文案。保存列配置返回失败时，前端不改写本地已确认记录，解除锁定并显示错误。查询令牌失效、薪资组或所属期变化时，保存请求前后均校验快照；失效结果不覆盖当前页面。

服务器事务是防止半写入的最终边界；前端锁仅用于避免重复提交和过期页面状态。

## 验证

- 单次点击保存列配置只产生一次浏览器 API 请求。
- API 成功后列配置记录完整返回，主页面无需二次列配置列表请求。
- API 失败时页面解锁，原列配置和预览不变。
- 查询、保存列配置、保存版式、打印准备期间，所有会改变快照的控件锁定且有加载文案。
- 页面离开提示仅在写入进行中出现。
- 同一保存操作不能被重复触发。

## 回滚

保留现有模型方法与列配置模型记录。若新 API 编排验证失败，前端可回退到当前按列保存实现；上线切换前不删除旧路径。
